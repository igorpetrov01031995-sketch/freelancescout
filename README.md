## 1. СУТЬ ПРОЕКТА (Зачем это вообще нужно)

**Проблема:** Фриланс-биржи публикуют десятки новых заказов в час.
Чтобы найти подходящий — нужно вручную читать каждый, оценивать реализуемость,
прикидывать стек, считать сроки. Это 1-2 часа ежедневной рутины.

**Решение:** Бот мониторит биржи сам. На каждый новый заказ запускает
консилиум из 4 ИИ-агентов и присылает готовый вердикт в Telegram:
брать проект или пропустить, и если брать — вот план и черновик ответа клиенту.

**Ценность на выходе:**
- Ты видишь только отфильтрованные заказы с готовым планом
- Экономия: 1-2 часа в день вместо ручного просмотра
- Скорость отклика: ты отвечаешь клиенту раньше конкурентов
- База данных лидов в Google Sheets для анализа рынка

## 2. КАК ЭТО РАБОТАЕТ (Поток данных)

```
[Kwork RSS / FL.ru RSS] ← проверка каждые 15 минут
         ↓
[Фильтр] → уже виденные заказы отсекаются через seen_ads.json
         ↓
[Новый заказ найден] → текст передаётся в AI-движок
         ↓
[Groq API / Llama 3.3 70b] → консилиум 4 агентов в одном запросе
         ↓
[Структурированный вердикт]
         ↓
    ┌────┴────┐
    ↓         ↓
[Telegram]  [Google Sheets]
 вердикт     строка лога:
 в личный    дата, биржа, заказ,
 чат         вердикт, скоринг
```

**Ручной режим (уже работает):**
Вставляешь текст любой задачи боту → получаешь вердикт за 3 секунды.

## 4. СПЕЦИФИКАЦИЯ ИИ-АГЕНТОВ

Все 4 агента работают внутри одного запроса к Groq.
Модель применяет роли поочерёдно и возвращает единый структурированный ответ.

---

### 🔍 Агент 1: СКАУТ-ОЦЕНЩИК
**Задача:** первичная разведка заказа за 10 секунд

**Анализирует:**
- Реален ли бюджет для данной задачи? (заниженный / адекватный / щедрый)
- Реален ли срок? (красный флаг если "всё за 2 дня" при сложном ТЗ)
- Есть ли признаки токсичного клиента? (правки бесконечные, "по-быстрому")
- Какой реальный объём работы скрыт за коротким описанием?

**Выдаёт:** Скоринг заказа от 1 до 10. Красные флаги если есть.

---

### 💻 Агент 2: ТЕХНИЧЕСКИЙ ДИРЕКТОР
**Задача:** разобрать как именно это делается

**Анализирует:**
- Какой стек технологий оптимален для задачи?
- Какие библиотеки / инструменты использовать?
- Главный технический риск проекта (что может пойти не так)
- Сложность по шкале 1-10 для разработчика уровня middle

**Выдаёт:** Конкретный стек + главный риск + оценка сложности.

---

### 📋 Агент 3: ПРОЕКТ-МЕНЕДЖЕР
**Задача:** декомпозировать задачу на шаги

**Анализирует:**
- Разбивка на этапы (3-5 конкретных шагов реализации)
- Реальная оценка времени на каждый этап
- Что нужно уточнить у клиента до старта (2-3 вопроса)
- Скрытые объёмы: что клиент не написал, но подразумевает

**Выдаёт:** Пошаговый план + вопросы клиенту + реальный дедлайн.

---

### ⚖️ Агент 4: АРБИТР (финальный вердикт)
**Задача:** принять решение и дать инструмент для отклика

**Выдаёт:**
- Вердикт: БРАТЬ / ПРОПУСТИТЬ (одно слово + одно предложение почему)
- Черновик первого сообщения клиенту (готово к отправке)
- Три первых действия если берём проект

## 9. ДЛЯ КОЛЛЕГ: ПРАВИЛА НАПИСАНИЯ КОДА

1. **Логи на каждом шаге:** console.log("Шаг 1: получаем RSS..."), console.log("Найдено заказов: X")
2. **Никогда не пиши всё целиком** — только один модуль за раз, тестируем, потом следующий
3. **Ошибки должны быть читаемыми:** не просто throw error, а console.error("Ошибка на шаге RSS:", error.message)
4. **Секреты только в .env** — никаких ключей в коде
5. **Комментарий перед каждой функцией** — одна строка что она делает
